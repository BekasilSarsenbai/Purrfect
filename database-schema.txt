# Purrfect - Database Schema Document

## 1) Database Design Goals
- Preserve financial correctness for escrow and split payouts.
- Enforce listing integrity and anti-fraud constraints.
- Support dispute evidence workflows and moderator decisions.
- Keep relational data strongly typed through Prisma models.

Database engine: PostgreSQL 15+  
ORM: Prisma ORM (no raw SQL in application business operations).

## 2) ERD (ASCII)
```text
users (1) --------< listings (N) --------< listing_media (N)
  |  \               |
  |   \              \------< listing_documents (N) >------ document_verifications (N)
  |    \
  |     \------< refresh_tokens (N)
  |
  \------< orders (as buyer) >------ listings (as item)
                     |
                     \------< escrow_transactions (N)
                     |
                     \------< inspections (0..1)
                     |
                     \------< disputes (0..1) ------< dispute_evidence (N)
                     |
                     \------< payouts (N)

users (1) --------< moderation_cases (N)
users (1) --------< audit_logs (N)
users (1) --------< notifications (N)
```

## 3) Primary Tables and Relationships
- `users`: root identity table.
- `refresh_tokens`: session/token rotation support; many tokens per user.
- `listings`: seller-owned cat offers with lifecycle statuses.
- `listing_media`: photos/videos per listing.
- `listing_documents`: uploaded pedigree/vaccine docs.
- `document_verifications`: moderator decisions for uploaded docs.
- `orders`: buyer-seller deal linked to one listing.
- `escrow_transactions`: atomic money movements tied to order.
- `payouts`: milestone settlement records.
- `inspections`: veterinary check results with 72-hour deadline.
- `disputes`: dispute case per order (max one active).
- `dispute_evidence`: file/text evidence for dispute case.
- `moderation_cases`: moderation queue and outcomes.
- `audit_logs`: append-only security/financial audit records.
- `notifications`: user communication events.

## 5) Key Constraints and Integrity Notes
- One active inspection per order (`Inspection.orderId @unique`).
- One dispute object per order (`Dispute.orderId @unique`); status handles lifecycle.
- Milestone payout uniqueness (`@@unique([orderId, milestone])`).
- Token rotation support with `familyId` and hashed token storage.
- Audit logs are append-only; update/delete operations on audit table are not exposed by service layer.

## 6) Migration Strategy (Prisma Migrate)
Migration process:
1. Modify `schema.prisma` and run migration generation.
2. Validate generated SQL in staging.
3. Apply expand-and-contract:
   - Expand: add nullable columns/new tables first.
   - Backfill data through controlled scripts.
   - Switch application reads/writes to new fields.
   - Contract: remove deprecated columns in later release.
4. Never drop critical columns in same release as code switch.
5. Financial table migrations require maintenance window and rollback plan.

Data loss prevention:
- mandatory backup before destructive migration,
- migration reviewed in PR with schema diff,
- canary rollout for production migration steps.

## 7) Nasty Query (Slowest Expected Query)
Use case: moderation dashboard needs suspicious listings by combined risk and dispute history, ordered by urgency.

Query (for analysis and index planning):
```sql
SELECT
  l.id,
  l.seller_id,
  l.breed,
  l.price_kzt,
  l.risk_score,
  COUNT(d.id) FILTER (WHERE d.status IN ('OPEN','UNDER_REVIEW')) AS active_disputes,
  MAX(d.opened_at) AS last_dispute_at
FROM listings l
LEFT JOIN orders o ON o.listing_id = l.id
LEFT JOIN disputes d ON d.order_id = o.id
WHERE l.status = 'PUBLISHED'
  AND l.risk_score >= 60
GROUP BY l.id
ORDER BY l.risk_score DESC, active_disputes DESC, l.created_at DESC
LIMIT 100;
```

Index strategy:
- `listings(status, risk_score, created_at DESC)`
- `orders(listing_id)`
- `disputes(order_id, status, opened_at DESC)`

Expected plan notes:
- Index scan on high-risk published listings.
- Hash/merge join from orders -> disputes.
- Group aggregation on listing id with filtered count.
- For high cardinality, partial index on `disputes` for active statuses can reduce heap reads.

Predicted EXPLAIN ANALYZE behavior (after warm cache):
- latency target < 120 ms for 500k listings / 5M orders scale.
- dominant cost in join+aggregate stage; mitigated by composite indexes and capped result window.

## 8) Cursor Pagination Standard
All list APIs use keyset pagination; canonical cursor fields:
- primary sort: `createdAt DESC`
- tie-break sort: `id DESC`

Reason: stable performance and consistency at large offsets versus OFFSET/LIMIT degradation.
